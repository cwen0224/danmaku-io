# 彈幕IO 開發準則（v0.1）

## 1. 專案目標
- 先完成「單手滑鼠操作」可玩核心：移動/瞄準/自動斬擊/敵人逼近。
- 架構優先於內容量：每個附屬系統必須可插拔、可停用、可替換。
- 任何新功能都不得破壞核心 60 FPS 體感（低規機至少穩定 60）。

## 2. 核心玩法定義（不可漂移）
- 玩家控制：以滑鼠為主（移動 + 朝向）。
- 攻擊模式：角色朝滑鼠角度進行斬擊（可有冷卻、範圍、擊退）。
- 敵人行為：持續生成並向玩家逼近。
- 勝負條件：
  - 失敗：玩家生命歸零。
  - 成長：透過時間、生存或擊殺逐步增壓。

## 3. 架構原則
- **模組邊界清楚**：每個系統只做一件事。
- **資料驅動優先**：敵人數值、波次、武器、掉落使用 JSON/TS config。
- **事件導向解耦**：系統間透過事件匯流排（Event Bus）溝通，避免互相直接呼叫。
- **固定更新節奏**：戰鬥與碰撞採固定 timestep；渲染可變。
- **先 MVP 再擴充**：先保證可玩，再加附屬系統。

## 4. 建議模組切分
- `core/`
  - `game-loop`：主循環、fixed update、delta 管理。
  - `scene`：場景狀態（menu / playing / gameover）。
- `systems/`
  - `input-system`：滑鼠輸入、方向向量。
  - `player-system`：移動、生命、無敵幀。
  - `combat-system`：斬擊判定、冷卻、傷害與擊退。
  - `enemy-system`：敵人 AI、追擊、接觸傷害。
  - `spawn-system`：刷怪節奏與難度曲線。
  - `collision-system`：命中框/受擊框。
  - `progression-system`：升級、天賦、局內成長。
- `content/`
  - `enemies`、`weapons`、`waves` 設定檔。
- `ui/`
  - HUD、傷害數字、暫停/結算面板。

## 5. 程式碼規範
- 語言：TypeScript（嚴格模式 `strict: true`）。
- 命名：
  - 檔名 `kebab-case`。
  - 類別/型別 `PascalCase`。
  - 變數/函式 `camelCase`。
  - 常數 `UPPER_SNAKE_CASE`。
- 函式長度：單函式原則上不超過 60 行；超過必拆。
- 禁止魔法數字：戰鬥數值必須集中在 config。
- 公開介面要有最小註解：用途、輸入、輸出、副作用。

## 6. 新增附屬系統的接入規範
- 新系統必須提供：
  - `init(context)`
  - `update(dt)`
  - `dispose()`
- 新系統不得直接依賴 3 個以上其他系統；超過表示耦合過高。
- 透過事件註冊（例如 `onEnemyKilled`, `onPlayerDamaged`）接入，而非修改核心戰鬥流程。
- 所有可調參數先進 config，避免硬編碼。

## 7. 效能預算
- 單幀更新預算（邏輯 + 渲染）目標 < 16.6ms。
- 同屏敵人上限先定義（例：200），超過用降載策略：
  - 簡化遠距敵人更新頻率。
  - 限制特效粒子數。
  - 批次處理碰撞。
- 每次加新系統需量測：平均 FPS、1% low FPS、GC 次數。

## 8. 測試與驗收
- 最低測試要求：
  - 斬擊命中判定單元測試。
  - 刷怪曲線與時間推進測試。
  - 玩家死亡/重開流程測試。
- 每個功能 PR 至少附：
  - 變更說明。
  - 風險點。
  - 測試結果（含未覆蓋區）。

## 9. Git 與版本節奏
- 分支：`main` 穩定、`dev` 整合、功能分支 `feat/*`。
- Commit 格式：`type(scope): summary`
  - `feat`, `fix`, `refactor`, `perf`, `test`, `docs`, `chore`
- 里程碑：
  - M1：可玩核心（移動/斬擊/刷怪/死亡）
  - M2：成長系統（升級/天賦）
  - M3：附屬系統（任務、裝備、事件）

## 10. Definition of Done（DoD）
- 功能可啟用/停用，不破壞核心流程。
- 有最小測試或可重現驗證步驟。
- 無新增 console error。
- 效能未明顯退化。
- 文件已更新（至少一段變更記錄）。
